# BUG-001 — Tras confirmar el correo, la protectora no entra al onboarding

## Descripción

En producción (con confirmación de email activada), al pulsar el enlace de verificación el callback deja la sesión creada pero redirige a la **home** (`/`). Como el gate de onboarding solo actúa al entrar en `/panel`, la protectora se queda en la home y **nunca ve el wizard de alta**. En local no se detectó porque la confirmación está desactivada (el registro va directo a `/panel`).

Además, aterrizar en la home tras verificar es una UX pobre: no hay señal de que la verificación fue bien.

## Contexto / impacto

Rompe el criterio de "entrada automática al wizard tras registrarse como protectora" de [[FEATURE-002]] en el flujo real de producción. Sin esto, ninguna protectora nueva completa el alta salvo que adivine la URL `/panel`. Bloquea el onboarding real.

## Plan de desarrollo

### Seguridad

- El callback ya crea la sesión con `exchangeCodeForSession` (PKCE). No se re-pide login (el usuario ya está autenticado). La redirección de destino sigue siendo solo interna (nunca a orígenes externos) — mantener esa comprobación.

### Frontend

- **Nueva página `/(auth)/(verificado)/correo-verificado/page.tsx`** con layout propio `(verificado)/layout.tsx` que usa `AuthSplit image="/images/auth-verified.jpg"` (mismo patrón que `(alta)` con `auth-register.jpg`) → pantalla partida con imagen a un lado y contenido al otro, consistente con login/registro.
- Copy: título "¡Correo verificado! ✓", subtítulo cálido, botón **"Continuar"**. Textos en `messages/es.json` (namespace `auth`).
- El botón "Continuar" enruta **por rol**: protectora → `/panel` (el gate del middleware la lleva al wizard `/panel/alta`); adoptante → `/`. La página es un Server Component que lee el rol del usuario y pasa el destino al botón (o el botón consulta y navega).

### Backend

- **`src/app/auth/callback/route.ts`**: tras `exchangeCodeForSession` sin error, si no viene un `next` explícito (o `next` es `/`), redirigir a `/correo-verificado` en vez de a `/`. Respetar `next` cuando venga con una ruta interna concreta (p. ej. flujos OAuth con destino).

### Tareas TDD

1. Test del callback: código válido → redirige a `/correo-verificado` (cuando no hay `next` específico); `next` interno concreto se respeta; código inválido → `/login?error=oauth`.
2. Test de la página `correo-verificado`: renderiza título/botón; el destino del botón es `/panel` para rol shelter y `/` para adopter (mockeando el cliente Supabase).
3. Verificación visual: la página usa el split con imagen (fallback a degradado si la imagen no existe).

### Dependencias

- [[FEATURE-002]] (`hecho`).
- Imagen `public/images/auth-verified.jpg` (la genera el usuario; hay fallback de degradado si falta).

## Criterios de aceptación / Casuística a cubrir

- [ ] Al confirmar el correo, la protectora ve una pantalla "¡Correo verificado!" partida con imagen (consistente con login/registro), no la home.
- [ ] El botón "Continuar" lleva a la protectora a `/panel` → wizard `/panel/alta`; al adoptante a la home.
- [ ] El usuario queda logueado tras confirmar (no se le vuelve a pedir login).
- [ ] La redirección solo admite rutas internas (no open redirect).
- [ ] Si la imagen `auth-verified.jpg` no existe, la pantalla usa el degradado de fallback sin romperse.
- [ ] Un `next` interno concreto (p. ej. de un flujo OAuth) se sigue respetando.
